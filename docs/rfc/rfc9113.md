## Coverage Details

RFC 9113 defines the HTTP/2 binary framing layer, stream multiplexing, flow control, and the full lifecycle of HTTP/2 connections and streams. TurboHttp implements the complete client-side protocol including SETTINGS negotiation, PING/PONG keep-alive, GOAWAY graceful shutdown, PUSH_PROMISE handling, and stream-level flow control.

## Frame Types

HTTP/2 frames use a 9-byte header: `length(24) + type(8) + flags(8) + stream(31)`.

| Frame Type | Class | Purpose |
|-----------|-------|---------|
| DATA | `DataFrame` | Request/response body |
| HEADERS | `HeadersFrame` | Headers (+ END_HEADERS / END_STREAM) |
| CONTINUATION | `ContinuationFrame` | Header block continuation |
| RST_STREAM | `RstStreamFrame` | Stream reset |
| SETTINGS | `SettingsFrame` | Connection parameters |
| PING | `PingFrame` | Keep-alive |
| GOAWAY | `GoAwayFrame` | Graceful shutdown |
| WINDOW_UPDATE | `WindowUpdateFrame` | Flow control |
| PUSH_PROMISE | `PushPromiseFrame` | Server push (client reads only) |

## Implementation Notes

- **Stream IDs:** `StreamIdAllocatorStage` allocates client-initiated stream IDs (1, 3, 5, …).
- **HPACK:** Headers are encoded/decoded via `HpackEncoder`/`HpackDecoder` with synchronised dynamic tables. See [RFC 7541](/rfc/rfc7541).
- **Connection preface:** `PrependPrefaceStage` injects the HTTP/2 connection preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`) on first connect.
- **Flow control:** Both connection-level and stream-level `WINDOW_UPDATE` are handled by `Http20ConnectionStage`.
- **Correlation:** `CorrelationHttp20Stage` matches responses to requests by stream ID.
- **Known failures:** 6 RFC9113 tests have pre-existing H2 concurrency timing issues.
