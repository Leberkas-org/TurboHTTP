## Implementation Details

### CacheControlParser (§5.2)

Parses all standard `Cache-Control` directives:

| Directive | Type | Description |
|-----------|------|-------------|
| `max-age` | Response | Freshness lifetime in seconds |
| `s-maxage` | Response | Shared cache lifetime (takes precedence over max-age) |
| `no-cache` | Both | Revalidate before serving |
| `no-store` | Both | Do not cache |
| `must-revalidate` | Response | Stale responses must be revalidated |
| `public` / `private` | Response | Cache visibility |
| `max-stale` | Request | Accept stale responses up to N seconds |
| `min-fresh` | Request | Require N seconds of freshness |

### CacheFreshnessEvaluator (§4.2)

Calculates freshness in priority order:

1. `s-maxage` directive
2. `max-age` directive
3. `Expires` header
4. Heuristic freshness (10% of `Last-Modified` age, capped)

### CacheValidationRequestBuilder (§4.3)

Generates conditional requests for stale cache entries:

- **`If-None-Match`** — when `ETag` is present in cached response
- **`If-Modified-Since`** — when `Last-Modified` is present
- **304 Not Modified merge** — updates cached entry headers and resets freshness

### HttpCacheStore (§3)

Thread-safe in-memory LRU cache:

- **Vary support:** Caches separate entries per unique request header combination specified in `Vary`
- **Invalidation:** Unsafe methods (POST, PUT, DELETE, PATCH) invalidate matching cache entries
- **Configurable capacity:** Set via `CachePolicy` in `TurboClientOptions`
