# Plan: RFC 9114 (HTTP/3) — 99% Client Compliance

## Introduction

RFC 9114 has 145 client-side MUSTs and is currently at 1.4% (2/145). This plan defines the complete HTTP/3 client implementation in ~38 atomic tasks across 8 phases. Target: >=99% (143/145).

**Key architecture insight**: TurboHttp's transport layer is already abstracted via `IClientProvider`. No `QuicConnectionStage` is needed — the existing `ConnectionStage`, `ClientByteMover`, `ClientState`, and actor hierarchy work unchanged. Only a `QuicClientProvider` and `QuicOptions` are needed alongside the HTTP/3 protocol layer.

**Dependencies**: plan_017 (QPACK/RFC 9204) is a prerequisite for Phase 4+.

## Goals

- RFC 9114 client compliance from 1.4% to >=99% (143/145)
- Complete HTTP/3 frame codec
- QUIC integration via `QuicClientProvider` implementing `IClientProvider`
- `Http30Engine` as `IHttpProtocolEngine` in the existing Engine demuxer
- ~280 new tests with RFC-tagged DisplayNames

## Transport Architecture (No QuicConnectionStage Needed)

The existing transport abstraction already supports QUIC with minimal changes:

```
IClientProvider (existing interface)
├── TcpClientProvider  → NetworkStream     (existing)
├── TlsClientProvider  → SslStream         (existing)
└── QuicClientProvider → QuicStream         (NEW — only new transport code)

Transport selection point: ClientManager.Handle() — 1 line change
Options selection point:   TcpOptionsFactory.Build() — ~10 lines
```

**What stays unchanged**: `ConnectionStage`, `ClientByteMover`, `ClientState`, `ConnectionHandle`, `ConnectionActor`, `HostPool`, `PoolRouter` — all transport-agnostic.

### TASK-001: QuicOptions Record

**Description:** As a library developer, I want a `QuicOptions` record extending `TcpOptions` so that QUIC-specific settings are available.

**Implementation**:
- New file: `src/TurboHttp/Transport/QuicOptions.cs`
  - `public record QuicOptions : TcpOptions` with QUIC-specific fields:
    - `TimeSpan IdleTimeout { get; init; } = TimeSpan.FromSeconds(30)`
    - `int MaxBidirectionalStreams { get; init; } = 100`
    - `int MaxUnidirectionalStreams { get; init; } = 3`
    - `List<SslApplicationProtocol> ApplicationProtocols { get; init; } = [new("h3")]`

**Acceptance Criteria:**
- [ ] QuicOptions inherits all TcpOptions properties (Host, Port, ConnectTimeout, etc.)
- [ ] QUIC-specific properties have sensible defaults
- [ ] Unit test verifies inheritance

**Tests** (new file: `src/TurboHttp.Tests/RFC9114/01_QuicOptionsTests.cs`):

### TASK-002: QuicClientProvider

**Description:** As a library developer, I want a `QuicClientProvider` implementing `IClientProvider` so that QUIC connections return a `Stream`.

**Implementation**:
- New file: `src/TurboHttp/Transport/QuicClientProvider.cs`
  - `public sealed class QuicClientProvider(QuicOptions options) : IClientProvider`
  - `GetStreamAsync()` creates `QuicConnection` via `System.Net.Quic`, opens bidirectional stream, returns `QuicStream` (which IS a `System.IO.Stream`)
  - `Close()` disposes connection
  - `RemoteEndPoint` from `QuicConnection.RemoteEndPoint`

**Acceptance Criteria:**
- [ ] Implements `IClientProvider` interface
- [ ] `GetStreamAsync()` returns a `Stream` (QuicStream)
- [ ] Connection uses ALPN "h3"
- [ ] SNI set from `options.Host`
- [ ] Certificate validation callback forwarded from options

**Tests** (new file: `src/TurboHttp.Tests/RFC9114/02_QuicClientProviderTests.cs`):

| DisplayName | Method |
|-------------|--------|
| `RFC9114-3.2-QCP-001: Provider implements IClientProvider` | `Should_ImplementInterface` |
| `RFC9114-3.2-QCP-002: GetStreamAsync returns Stream` | `Should_ReturnStream_When_Connected` |
| `RFC9114-3.1-QCP-003: SNI extension sent` | `Should_SendSNI_When_Connecting` |
