# Plan: RFC 9110 (HTTP Semantics) — 99% Client Compliance

## Introduction

RFC 9110 has 99 client-side MUSTs. After plan_011 Tier 1-2 (TASK-001-009), 92 are covered. This plan defines the 6 additional tasks needed to reach 99% (98/99). These are added to plan_011 as TASK-029 through TASK-034.

## Goals

- RFC 9110 client compliance from 73% to 99% (98/99)
- Close all 18 MISSING items
- Re-evaluate and implement all 51 DEFERRED items where needed
- Remaining 1 item: §9.3.8.a TRACE sensitive data (only the application can decide what is sensitive)

## Preconditions

- **Userinfo stripping**: `UriSanitizer.cs` exists in `src/TurboHttp/Protocol/RFC9110/` with `FormatAuthority()`, `StripUserInfo()`, `FormatAbsoluteWithoutUserInfo()`
- **Referer sanitization**: `RequestEnricherStage.SanitizeReferer()` method exists in `src/TurboHttp/Streams/Stages/Routing/RequestEnricherStage.cs`
- **Certificate validation**: `TurboClientOptions.DangerousAcceptAnyServerCertificate` property exists
- **If-Range validation**: `IfRangeValidator.cs` exists in `src/TurboHttp/Protocol/RFC9110/`
- **CONNECT body handling**: `Http11Decoder` treats CONNECT 2xx body length as 0
- **206 validation**: `PartialContentValidator.cs` exists in `src/TurboHttp/Protocol/RFC9110/`

## User Stories

### TASK-001: Expect 100-Continue BidiStage

**Description:** As a library user, I want TurboHttp to automatically send `Expect: 100-continue` for large request bodies so that unnecessary uploads are avoided.

**RFC**: 9110 §10.1.1 — "Client waiting for 100 (Continue) MUST send Expect: 100-continue"

**Implementation**:
- New file: `src/TurboHttp/Protocol/RFC9110/Expect100Policy.cs`
  - `record Expect100Policy(long MinBodySizeBytes = 1024)`
- New file: `src/TurboHttp/Streams/Stages/Features/ExpectContinueBidiStage.cs`
  - Port names: `Expect100.In.Request`, `Expect100.Out.Request`, `Expect100.In.Response`, `Expect100.Out.Response`
  - Request direction: If body >= threshold → set Expect header, hold body
  - Response direction: 100 Continue → release body; 417 → abort; other → forward
- File: `src/TurboHttp/Streams/PipelineDescriptor.cs` — new field `Expect100Policy?`
- File: `src/TurboHttp/Streams/Engine.cs` — stack between Retry and Cache

**Acceptance Criteria:**
- [x] Requests with body >= threshold get `Expect: 100-continue` header
- [x] Body only sent after 100 response received
- [x] 417 Expectation Failed → request aborted, response forwarded
- [x] Small bodies (< threshold) pass through unchanged
- [x] Null policy → pure pass-through
- [x] Unit tests written and passing

**Tests** (new file: `src/TurboHttp.Tests/RFC9110/08_ExpectContinueTests.cs`):

| `RFC9110-10.1.1-EX-003: No body passes unchanged` | `Should_NotAddExpect_When_NoBody` |
| `RFC9110-10.1.1-EX-004: Request without content must not have Expect` | `Should_NotAddExpect_When_NoContent` |

**Stream-Test** (new file: `src/TurboHttp.StreamTests/RFC9110/07_ExpectContinueBidiStageTests.cs`):

|-------------|--------|
| `RFC9110-10.1.1-EBIDI-001: 100 response releases body` | `Should_ReleaseBody_When_100Continue` |

## Functional Requirements

- FR-1: MUST send `Expect: 100-continue` for large request bodies (§10.1.1)
- FR-2: MUST correctly parse WWW-Authenticate multi-challenge (§11.2.1)
- FR-3: MUST generate auth parameters as quoted-string (§11.2.2)
- FR-4: MUST generate Date header in IMF-fixdate when missing (§5.6.7)
- FR-5: MUST accept large decimal numerals in Range headers (§14.1.1)
- FR-6: MUST set Content-Encoding when compressing request bodies (§8.4)
- FR-7: MUST always send explicit port in CONNECT request-target (§9.3.6)

## Non-Goals

- Credential storage / session management (application responsibility)
- Automatic 401-retry with credentials (could be future AuthBidiStage)
- TRACE sensitive-data detection (not automatable — §9.3.8.a remains the 1 missing MUST)
