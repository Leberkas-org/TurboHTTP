MAJOR ARCHITECTURE CHANGES
===========================

FROM HTTP/2 TO HTTP/3:

1. TRANSPORT
   TCP + TLS to QUIC (UDP + TLS 1.3)
   New I/O layer required (System.Net.Quic)

2. HEADER COMPRESSION
   HPACK (RFC 7541) to QPACK (RFC 9204)
   New codec, separate unidirectional streams

3. STREAM MODEL
   Implicit stream 0 to Explicit unidirectional control stream per endpoint
   New stream type field (0x00=control, 0x01=push)

4. CONTROL MESSAGES
   SETTINGS on stream 0 to SETTINGS on control stream
   GOAWAY carries bidi stream ID (not push ID)
   NEW: MAX_PUSH_ID (client gates push), CANCEL_PUSH

5. PUSH SEMANTICS
   Implicit push ID to Explicit push ID space (0 - MAX_PUSH_ID)
   Server cannot push until client sends MAX_PUSH_ID

6. CERTIFICATE VALIDATION
   Per-connection to Per-origin validation
   Multi-origin reuse requires cert re-validation

CRITICAL CLIENT RESPONSIBILITIES
==================================

1. Send MAX_PUSH_ID first (gates server push)
2. Validate certificates per origin
3. Create control stream and send SETTINGS
4. Enforce GOAWAY monotonic stream IDs
5. Validate pseudo-headers (request-only vs response-only)
6. Implement 0-RTT stored settings compliance
7. Handle push stream duplicate detection
8. Reject non-authoritative push origins
9. Strict frame type/stream validation
10. Respect SETTINGS limits (e.g., MAX_HEADER_LIST_SIZE)

SUMMARY
=======

RFC 9114 defines HTTP/3, a complete HTTP protocol redesign over QUIC transport.
For TurboHttp, this requires:
  - New I/O layer (QUIC instead of TCP)
  - New codec (QPACK instead of HPACK)
  - New stream model (explicit control streams)
  - New control semantics (push ID space, certificate per-origin)

Implementation represents 15-23 weeks of development for a production-quality
HTTP/3 client. The current architecture can support this with Akka.Streams
integration and System.Net.Quic binding.

All 145 client-side MUST requirements have been extracted, categorized, and
documented for RFC compliance verification.
